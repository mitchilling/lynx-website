# lynx-ui 是什么

lynx-ui 将是你接入 Lynx 元件的第一站。我们是 Lynx 团队长期维护的 Headless UI 组件库。**Headless 是为了提供足够灵活、通用、高性能的 UI 解决方案**。

我们想用前端组件补足 native 元件的适配能力，建设高性能、native-like、兼容性好的 Lynx 组件生态。

同平台的 UI 特性在行为、API
乃至设计理念上都有显著差异，尤其是高阶功能更是如此。跨平台框架需要尽可能地收敛这些差异，Lynx
也不例外。

前端组件将在这些底层原子化 API 庞大的数量基础上，梳理它们的表现和限制，在前端侧达到最终的一致。

## 快速开始

lynx-ui 支持按组件引入和整体引入。

### 整体引入（推荐）

建议直接整体引入 `lynx-ui`。lynx-ui 支持 treeshaking，没有使用的组件不会占用体积。

<PackageManagerTabs command="install @lynx-js/lynx-ui" />

在项目中使用

```jsx {1}
import { Button } from '@lynx-js/lynx-ui';

export default function App() {
  return (
    <view>
      <Button />
    </view>
  );
}
```

### 按组件引入

lynx-ui 每个组件都有单独的包。如遇到兼容性或特定需求，可以只引入需要使用的组件。

这里以 `<Button>` 组件为例

<PackageManagerTabs command="install @lynx-js/lynx-ui-button" />

在项目中使用

```jsx {1}
import { Button } from '@lynx-js/lynx-ui-button';

export default function App() {
  return (
    <view>
      <Button />
    </view>
  );
}
```

:::details 我应该选择按组件引入还是整体引入
我们推荐整体引入，这样可以带来一些好处

1. 安装和使用更简单
2. treeshaking 会自动优化产物体积
   :::

## 配置

```typescript
export default defineConfig({
  ...
  plugins: [
    pluginReactLynx({
      ...
      targetSdkVersion: '2.14',
      enableNewGesture: true,
    }),
  ],
})
```

:::details 担心配置改动影响其他页面？
这里是整个组件库的配置需求，如果你的担心配置变动的影响，可以看看你使用的具体组件，配置需求会更宽松

:::

## 兼容性

- LynxSDK：>= 3.2

:::details 最低版本不满足需求？
这里是整个组件库的最低版本需求，如果你的最低版本不能满足需求，可以看看你使用的具体组件，版本需求会更宽松。
:::
